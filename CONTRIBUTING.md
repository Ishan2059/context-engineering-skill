# Contributing to Context Engineering Skill

First off, thank you for considering contributing to this project! 🎉

Context engineering is an evolving field, and community contributions help keep this skill current, comprehensive, and valuable for everyone.

## 🤝 How Can I Contribute?

### 1. Share Your Experience
- Open an issue describing how you used the skill
- Share results and metrics from your implementation
- Suggest improvements based on your use case

### 2. Report Issues
- Found a typo or error? Let us know!
- Unclear instructions? Help us improve them
- Missing information? Suggest what to add

### 3. Suggest Enhancements
- New context engineering patterns
- Additional tools and technologies
- More use cases and examples
- Better explanations or analogies

### 4. Submit Pull Requests
- Fix bugs or typos
- Add new techniques or patterns
- Improve documentation
- Add examples and code snippets

### 5. Help Others
- Answer questions in Issues
- Share the skill with your network
- Write blog posts or tutorials
- Create video content

## 📝 Contribution Guidelines

### For SKILL.md Updates

When contributing to the main skill file:

1. **Maintain Structure**: Keep the existing organization (principles, patterns, techniques)
2. **Be Concise**: Claude reads this file, so clarity and brevity matter
3. **Cite Sources**: If referencing research or specific techniques, include attribution
4. **Test First**: Try your addition with Claude before submitting
5. **Stay Practical**: Focus on actionable, proven techniques

### For Documentation Updates

When improving README, guides, or other docs:

1. **Clear Language**: Write for diverse audiences (beginners to experts)
2. **Examples**: Include concrete examples when possible
3. **Formatting**: Use markdown consistently
4. **Links**: Ensure all links work and are relevant
5. **Accuracy**: Verify technical details

### Code of Conduct

- Be respectful and inclusive
- Welcome newcomers and help them contribute
- Focus on constructive feedback
- Assume good intentions
- Keep discussions professional and on-topic

## 🔄 Pull Request Process

1. **Fork** the repository
2. **Create a branch** for your changes (`git checkout -b feature/amazing-pattern`)
3. **Make your changes** with clear, descriptive commits
4. **Test** your changes (install the skill and verify it works)
5. **Update documentation** if needed
6. **Submit a pull request** with a clear description

### PR Description Template

```markdown
## What does this PR do?
Brief description of the changes

## Why is this needed?
Explain the problem or opportunity

## How was it tested?
Describe how you verified the changes work

## Additional context
Any other relevant information
```

## 🐛 Reporting Bugs

When reporting bugs, please include:

1. **Description**: What's wrong?
2. **Steps to Reproduce**: How can we see the issue?
3. **Expected Behavior**: What should happen?
4. **Actual Behavior**: What actually happens?
5. **Environment**: Claude version, platform, etc.
6. **Screenshots**: If applicable

## 💡 Suggesting Features

When suggesting new features or content:

1. **Use Case**: Describe the scenario
2. **Current Gap**: What's missing now?
3. **Proposed Solution**: What should be added?
4. **Benefits**: Who does this help and how?
5. **Examples**: Show what it would look like

## 📚 Style Guide

### Writing Style
- Use clear, direct language
- Write in present tense
- Use active voice
- Avoid jargon unless necessary (then explain it)
- Use examples to illustrate concepts

### Formatting
- Use markdown headers hierarchically (# → ## → ###)
- Use code blocks for code, commands, and file contents
- Use bullet points for lists
- Use tables for comparisons
- Use bold for emphasis, not italics

### Technical Content
- Cite sources for research or specific claims
- Include version numbers for tools/frameworks
- Provide working code examples
- Test all commands and code before submitting
